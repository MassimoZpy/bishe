__传统开源捐赠平台有效性的分析评估  
Open Collective 与 GitHub Sponsors 的对比研究__

南京大学本科毕业论文草稿

# 摘要

开源软件已经从技术协作实践演变为数字社会的重要基础设施。企业生产系统、科研协同平台、公共部门数字化系统以及人工智能开发链路普遍依赖开源组件，但价值创造与价值回流之间长期存在结构性错配：需求侧持续获得高额收益，维护侧却仍以无偿或低偿劳动为主。该矛盾不仅影响项目可持续性，也关系到软件供应链安全、漏洞响应效率和生态治理稳定性。 围绕"平台化捐赠是否能够有效改善开源项目可持续性"这一问题，本文以 GitHub Sponsors 与 Open Collective 为研究对象，在梳理平台规则、学术研究和行业报告的基础上，构建了转化、稳定、治理、公平四维评估框架。本文进一步提出以项目\-\-月份为单位的准实验研究路径，结合倾向得分匹配、双重差分、事件研究和分位数回归等方法，为后续实证检验提供可执行方案。 研究认为，GitHub Sponsors 的优势在于低摩擦入口与工作流内嵌带来的短期转化效率，Open Collective 的优势则在于 Fiscal Host、公开账本和报销机制所提供的治理与合规能力。两类平台并非简单替代关系，而是在项目生命周期中具有明显互补性。若缺乏预算披露、角色分配和赞助沟通等内部制度，平台带来的流量与资助冲击难以转化为长期稳定收入。相反，平台工具与项目治理能力形成协同时，更有可能将偶发捐赠沉淀为可持续维护能力。 本文的主要贡献包括：第一，将平台比较从单纯的金额和人数比较扩展为转化、稳定、治理、公平四维综合评估；第二，提出兼顾可操作性与可复核性的研究设计，为后续数据验证提供了清晰路径；第三，从维护者、财政托管机构、企业、平台治理者和公共部门等多个主体出发，提出将开源资助从"偶发捐赠逻辑"升级为"基础设施治理逻辑"的政策建议。本文也指出，跨平台数据口径不一致、平台采用存在内生性以及跨法域差异等问题，仍需在未来研究中进一步识别。

# Abstract

Open\-source software has evolved from a collaborative development practice into a critical infrastructure of digital society\. Enterprise systems, research platforms, public\-sector digital services, and AI development pipelines all depend heavily on open\-source components\. Yet a persistent structural mismatch remains between value creation and value capture: demand\-side actors receive enormous benefits, while maintainers still rely largely on unpaid or underpaid labor\. This mismatch threatens project sustainability and affects software supply\-chain security, vulnerability response, and ecosystem governance\. This thesis examines whether platform\-based donations can effectively improve the sustainability of open\-source projects\. Focusing on GitHub Sponsors and Open Collective, it reviews platform rules, academic studies, and industry reports, and proposes a four\-dimensional framework covering conversion, stability, governance, and equity\. Based on that framework, the thesis further develops a quasi\-experimental empirical roadmap using project\-month panels, propensity score matching, difference\-in\-differences estimation, event studies, and quantile regression\. The analysis suggests that GitHub Sponsors performs strongly in short\-term donor conversion because of its low\-friction workflow integration, whereas Open Collective performs better in governance and compliance because of its fiscal\-host model, public ledgers, and reimbursement processes\. The two platforms should therefore be understood as complementary rather than mutually exclusive\. Without internal mechanisms such as budget disclosure, role\-based allocation, and sponsor communication, short\-term donation shocks are unlikely to become stable long\-term support\. When platform tools are combined with stronger project governance, however, ad hoc donations can more plausibly be transformed into sustainable maintenance capacity\. This thesis contributes in three ways\. First, it moves platform comparison beyond simple funding totals toward a broader evaluation of conversion, stability, governance, and equity\. Second, it offers a concrete and reproducible research design for later empirical validation\. Third, it develops policy recommendations for maintainers, fiscal hosts, enterprises, platform operators, and public actors, arguing that open\-source funding should be treated as an infrastructure\-governance issue rather than merely a product\-choice problem\. The thesis also acknowledges current limitations, including inconsistent cross\-platform data, endogeneity in platform adoption, and differences across legal and cultural environments\.

# 绪论

## 研究背景

开源软件已经成为数字经济和公共数字基础设施的重要组成部分。无论是云计算平台、数据库、中间件、科研协同工具，还是人工智能模型训练与部署链路，都建立在大量开源依赖之上。已有研究指出，若企业和组织需要在失去开源生态支持的情况下自行重建同类能力，将面临极高的替代成本，这表明开源软件在社会层面创造了可观的外部价值\[13\]。与此同时，维护者群体却常常承受高责任、低回报的压力，资金支持与劳动投入长期不匹配。

这种错配并不仅仅是个人收入问题，更关乎项目可持续性和供应链安全。维护者需要承担漏洞修复、版本兼容、文档更新、社区协调、代码审查等持续性工作，但这些劳动中相当一部分并不能稳定转化为收入。Tidelift 的维护者调查也反映出，维护者普遍面临时间不足、收入不稳定以及职业路径模糊等问题\[14\]。因此，如何建立更可持续的开源资助机制，已经从社区内部议题上升为与公共治理、产业安全和数字基础设施韧性相关的问题。

## 问题提出

近年来，平台化捐赠逐渐成为开源项目获得资金支持的重要方式。其中，GitHub Sponsors 依托代码托管平台，提供低门槛、低摩擦的赞助入口\[1, 2\]；Open Collective 则通过 Fiscal Host、公开账本和报销流程，为项目提供更加制度化的财务治理基础设施\[5, 6\]。这两类平台在资助实践中都具有代表性，但它们所解决的问题并不完全相同。

现有讨论往往聚焦于"哪个平台更容易筹到钱"、"哪个平台赞助人数更多"之类的表层比较。这类问题固然重要，却不足以回答更关键的治理问题：资金是否稳定、是否可审计、是否能覆盖非代码贡献、是否会进一步强化头部项目优势。若只以金额和人数为尺度，可能高估短期流量冲击的作用，而忽视了治理能力不足所带来的中长期风险。

基于此，本文关注的核心问题不是简单比较两个平台的产品功能，而是分析平台机制如何影响开源项目的长期可持续性。换言之，本文试图从"平台比较"进一步推进到"资助制度设计"。

## 研究对象与研究边界

本文的研究对象为 GitHub Sponsors 与 Open Collective。之所以选择这两个平台，一方面是因为二者在开源生态中具有较高知名度和实践代表性，另一方面则是因为它们在机制设计上形成了鲜明对比。GitHub Sponsors 更强调开发流程内嵌、赞助路径便捷和赞助触达效率；Open Collective 更强调资金流透明、支出可追踪、团队报销与合规治理。正是这种机制差异，为比较研究提供了清晰切口。

本文在研究边界上作出以下限定。第一，本文主要使用公开可核验的平台规则、公开研究和项目资料，不使用无法追溯来源的所谓"内部数据"。第二，本文重点关注开源项目的可持续资助与治理机制，不对平台商业模式作全面产品评测。第三，本文提出的实证部分属于可复现研究设计与证据综合框架，旨在为后续数据验证提供路径，而非伪造大样本统计结果。这样的边界设置有助于保证研究的规范性与可复核性。

## 研究意义

从理论层面看，本文尝试将开源资助研究从"金额增长"的单一维度推进到"转化、稳定、治理、公平"四维综合评估，回应了开源可持续性研究中长期存在的指标单一问题。平台是否有效，并不只体现在短期筹资数额上，还取决于其是否有助于形成更稳定、更可解释、更具公平性的制度安排。

从实践层面看，本文希望为不同主体提供可操作的分析框架。对维护者而言，研究能够帮助其理解何时更应重视赞助入口效率，何时更应补强预算透明与团队分配规则。对企业和开放源代码项目办公室而言，研究能够提示其将资助纳入供应链风险治理，而不是将其视为单纯的品牌或公益活动。对平台和财政托管机构而言，研究也提供了思考如何改善中尾部项目支持、如何提升治理工具可用性的参考。

## 研究思路与论文结构

本文首先通过文献与平台规则梳理，总结现有研究与制度安排的主要发现；其次构建转化、稳定、治理、公平四维分析框架，并在此基础上提出研究问题与研究假设；再次从机制层面对 GitHub Sponsors 与 Open Collective 的制度逻辑进行比较；随后结合公开研究、行业报告和平台规则，对两平台在转化、治理和分配结构上的表现进行综合讨论；最后提出面向多类主体的治理与政策建议，并说明本文研究的局限与未来拓展方向。

全文共分为八章。第一章为绪论，说明研究背景、问题、对象与意义；第二章回顾相关文献和理论基础；第三章提出分析框架与研究设计；第四章比较两类平台的机制差异；第五章对现有证据进行综合分析并展开讨论；第六章提出治理与政策建议；第七章总结研究局限并提出未来研究方向；第八章给出全文结论。

# 文献综述与理论基础

## 平台规则与制度差异的既有描述

围绕 GitHub Sponsors 与 Open Collective 的比较，最直观的资料来自平台自身公开的制度文档。GitHub Sponsors 的官方说明强调其对开发工作流的嵌入性，支持个人或组织通过 GitHub 账户直接建立持续性或一次性赞助关系，并在一定条件下为开发者侧资金转入降低手续摩擦\[1, 3\]。这类设计有助于提升潜在赞助者的进入效率，尤其适合已经在 GitHub 生态内拥有较高可见度的项目或个人维护者。

相比之下，Open Collective 的官方文档更强调 Fiscal Host、公开账本、预算披露和报销处理等组织化能力\[5, 4, 6\]。项目在获得资金的同时，也被鼓励建立更明确的预算说明与支出流程。公开资料还显示，Open Source Collective 等 Host 会公开费用结构与托管规则\[7\]。这说明 Open Collective 所解决的问题并不局限于"如何收钱"，更包括"如何解释和管理钱"。

## 关于资助行为与平台效应的实证研究

现有研究已经开始从经验层面对开源资助平台进行分析。有关 GitHub Sponsors 的研究表明，赞助机制对开发者行为和社区互动可能产生短期正向影响，但这种影响往往较为温和，且未必能够自然延续为长期稳定效应\[12, 9, 10\]。这提示我们，平台降低进入门槛并不等于项目一定获得可持续支持，后续仍需制度安排承接资助关系。

对于 Open Collective，相关研究则更多关注捐赠类型、捐赠者行为差异以及公开账本所反映出的治理特征。已有工作指出，个人捐赠者和组织捐赠者在金额、复捐黏性和项目支持模式上存在明显差异，且不少项目的支出结构并不限于代码开发，还包括文档、运营、社区与其他非工程性劳动\[8\]。这些发现说明，若只考察筹资总额，很容易忽略资助结构和支出角色差异对项目治理的重要影响。

此外，关于社交媒体传播与资助结果之间关系的研究也揭示出平台效应的外部条件。Munaiah 等人的研究表明，开发者在 Twitter/X 上的可见性提升可能带来 GitHub Sponsors 的短期冲击，但并非所有发起资助的开发者都能有效获得资金，说明传播只是影响转化的条件之一，而非唯一决定因素\[11\]。

## 理论基础

本文主要借鉴四类理论资源。第一是公共品与集体行动理论。开源软件具有显著的非排他性和外部性，受益者广泛分散，而资助责任往往难以自动落实，这使得"搭便车"问题在开源生态中尤为突出。第二是平台治理理论。平台并不仅仅是中介，它通过规则、界面与激励机制塑造行为边界，从而影响参与者之间的互动方式和资源流动方向。

第三是组织制度理论。透明度、问责链条、角色分工和合法性机制等制度要素，会显著影响合作关系的稳定性和组织外部信任。对于开源项目而言，资助能否真正转化为维护能力，并不只取决于资金规模，还取决于项目是否拥有足够清晰的预算结构、公开说明和团队分配规则。第四是注意力经济与马太效应理论。平台上的可见度、品牌识别和网络关系会强化资源向头部项目集中，这使得中尾部项目即便同样重要，也未必能够获得与其基础设施地位相匹配的支持。

## 现有研究的不足

综观上述研究，可以发现现有文献仍存在三方面不足。第一，许多讨论过度聚焦筹资结果，而对治理透明度、团队分配公平性和非代码劳动补偿关注不足。第二，平台采用本身往往存在选择偏差，优质项目更有可能获得曝光和赞助，因此若缺乏较好的识别策略，很难区分平台效应与项目自身成长之间的关系。第三，现有研究较少从项目生命周期与多主体治理的角度讨论平台协同策略，容易将平台比较理解为单一产品竞赛。

## 本文的研究切入点

基于上述不足，本文选择从机制比较和制度设计两个层面切入。其一，本文不以"哪个平台更好"作为唯一判断目标，而是强调不同平台在不同维度上的优势和边界。其二，本文构建转化、稳定、治理、公平四维分析框架，以便将制度设计问题具体化、指标化。其三，本文给出准实验研究路径，将平台规则和公开研究中的经验性发现组织进一个可复现、可扩展的分析框架中。通过这一切入点，本文希望把开源资助讨论从经验观察推进到更具解释力的学术分析。

# 分析框架与研究设计

## C\-S\-G\-E 四维分析框架

为了从更完整的角度评估平台化资助对开源项目可持续性的影响，本文提出转化（Conversion）、稳定（Stability）、治理（Governance）和公平（Equity）四维分析框架，简称 C\-S\-G\-E 框架。该框架试图同时回答四个问题：平台是否有助于吸引新的赞助者，资助关系是否能够稳定延续，资金流是否能够被解释和管理，分配结果是否覆盖关键角色并避免过度集中。

其中，转化维度主要考察赞助入口效率，如首赞转化、赞助层级设置、赞助发起便利性和新增赞助者增长。稳定维度主要考察资助关系的持续性，如复捐率、收入波动、冲击后回落情况和连续正收入月占比。治理维度关注预算公开、支出可追踪、报销规范和合规支撑。公平维度则关注资金集中度、角色覆盖情况以及中尾部项目是否有可能形成基本保障。

四个维度之间并非相互独立。一个平台即便具备较高转化效率，也可能因为治理能力不足而难以沉淀长期支持；反之，一个平台即便治理工具完善，也可能因为进入门槛较高而不利于早期项目建立第一批支持者。因此，评价平台有效性不能只依赖单一指标，而应结合不同维度之间的相互作用进行综合判断。

## 研究问题（RQ）与研究假设

为与后续实证部分严格对齐，本文将研究问题收敛为三个可操作的 RQ（Research Questions），并在每个 RQ 下配置可检验假设与实验方案。

**RQ1（转化与持续性）**：在控制项目基础活跃度后，采用 GitHub Sponsors 或 Open Collective 是否会显著提升项目的短期赞助转化与中期留存稳定性？两平台效应是否存在阶段性差异？

- H1a：接入 GitHub Sponsors 后，项目在 0~3 个月窗口更可能出现新增赞助者的显著上升。
- H1b：若缺乏预算披露与周期性进展反馈，GitHub Sponsors 的短期冲击在 6~12 个月将明显衰减。
- H1c：同时配置 Open Collective 治理后端的项目，其中期收入波动系数更低。

**RQ2（治理机制与信任）**：Fiscal Host、公开账本与报销可追踪机制，是否能提高项目财务透明度与外部信任，并进一步提升复捐概率？

- H2a：采用 Fiscal Host 的项目在“预算公开完整度、支出可追溯率、角色覆盖率”三项治理指标上显著更高。
- H2b：治理透明度指数（GTI）每提升一个标准差，项目复捐率将呈正向变化。
- H2c：治理机制对组织捐赠者（机构赞助者）的稳定作用强于个人捐赠者。

**RQ3（分配公平与结构性偏差）**：平台化捐赠是否会加剧头部集中？若是，哪些治理变量可缓解中尾部项目“可见度不足但基础设施关键性高”的资金缺口？

- H3a：不论平台类型，资金分配都存在显著头部集中（长尾）特征。
- H3b：公开预算、角色化报销、长期资助承诺比例上升，可显著改善中位项目的资金增速。
- H3c：平台转化能力与治理能力存在互补关系，“前端转化+后端治理”策略可降低结构性不平等。

## 样本构建与变量设计

若将本文设计进一步落实为项目\-\-月份面板数据，可优先选择近两年内持续活跃、具有公开资助入口且具备一定社区可见度的开源项目作为研究样本。项目可按平台采用状态划分为仅使用 GitHub Sponsors、仅使用 Open Collective、同时使用两平台以及暂未采用平台的对照组。观察窗口可设置为平台采用前后 24 至 36 个月，以捕捉接入前后变化与可能存在的季节性波动。

变量设计与四维框架相对应。转化类变量包括新增赞助者数量、赞助金额中位数和赞助层级变化；稳定类变量包括复捐率、月收入波动系数和连续正收入月占比；治理类变量包括预算公开频率、支出说明完整度和账本透明度指数；公平类变量包括 Gini 系数、前十大项目份额、中位项目资金增长率和非代码角色覆盖情况。除了这些核心变量外，还需要控制项目年龄、star 数量、贡献者规模、release 频率和 issue 响应速度等基础特征，以降低项目内生差异带来的偏误。

## 识别策略与模型设定

为了尽可能区分平台采用效应与项目自身成长效应，本文提出将倾向得分匹配与双重差分方法结合使用。首先，基于项目在平台采用前的可观测特征进行匹配，使处理组和对照组在基础规模和活跃度上更具可比性；其次，在匹配样本上估计平台采用的平均处理效应。基本形式可以写为：

Y\_\{it\} = \\alpha \+ \\beta \(Treat\_i \\times Post\_t\) \+ \\gamma X\_\{it\} \+ \\mu\_i \+ \\lambda\_t \+ \\varepsilon\_\{it\}

其中，Y\_it 表示项目 i 在时期 t 的结果变量，Treat\_i 表示项目是否属于处理组，Post\_t 表示平台采用后的时期，X\_it 为控制变量，mu\_i 和 lambda\_t 分别表示项目固定效应和时间固定效应。系数 beta 即为本文关注的平台采用效应。

在此基础上，还可以结合事件研究方法考察平台采用前后的动态路径，并使用分位数回归识别平台效应在不同资金层级项目中的异质性。如果高分位项目受益显著，而中位和低位项目改善有限，则说明平台可能强化了头部集中结构。稳健性检验则包括平行趋势检验、虚拟采用月安慰剂检验、替代因变量检验和极端值处理等。

## 研究伦理与规范性说明

本文当前版本主要依托公开规则、公开研究与公开数据样例开展分析，不涉及对个人敏感信息的抓取和披露。若未来扩展到更细粒度数据收集，应进一步遵循数据最小化、匿名化和知情同意等研究伦理原则。与此同时，本文明确区分"基于公开证据的综合推断"与"完成大样本统计检验后的因果结论"。这种区分既是学术规范要求，也是避免伪造经验结果的重要前提。

# 平台机制比较：制度逻辑与行为后果

## GitHub Sponsors：低摩擦入口机制

GitHub Sponsors 的制度优势首先体现在其进入路径与开发工作流的高度耦合。开发者与项目参与者本身就在 GitHub 平台内进行代码托管、协作、问题跟踪与社交互动，因此赞助功能嵌入到仓库页面、个人主页和组织界面之后，潜在赞助者可以在几乎不离开现有工作流的情况下完成资助动作\[1\]。这种机制显著降低了从"看到项目"到"发起支持"之间的操作阻力，也更容易把用户对项目的即时好感转化为实际赞助。

此外，GitHub Sponsors 支持月度赞助与一次性赞助，并兼顾个人和组织发起资助的场景\[2, 3\]。对早期项目或个人维护者而言，这种设置尤其具有吸引力，因为其不要求项目一开始就具备复杂的财务治理能力。只要维护者已经在 GitHub 上建立一定可见度，便能迅速测试资助意愿并尝试积累第一批支持者。

但 GitHub Sponsors 的边界同样明显。首先，它更擅长帮助项目"把钱引进来"，却不会自动回答"钱如何解释、如何分配、如何形成长期预算"等问题。其次，若项目面对的是团队协作、多角色支出或较强的合规要求，仅依赖赞助入口本身并不能构成完整治理方案。最后，当平台主要优化首赞动作而缺少对预算透明和治理反馈的支持时，项目很容易获得短期冲击，却难以维持稳定关系。

## Open Collective：治理基础设施机制

与 GitHub Sponsors 相比，Open Collective 的制度逻辑更偏向治理与组织化协作。通过 Fiscal Host 模式，项目可以将收入接收、财务托管、报销审核和合规处理嵌入到一个较为完整的制度安排之中\[5\]。平台鼓励项目公开展示收入、支出和用途说明，使赞助者、社区成员和外部观察者能够更直接地理解资金流向。这种透明性本身就构成了一种治理信号。

公开账本和报销机制的另一项重要意义，是它使非代码劳动更容易被制度化地看见。文档编写、翻译、社区运营、设计、活动组织和安全应急等工作，往往难以像代码提交那样通过平台行为直接体现，但在 Open Collective 的支出与预算逻辑中，这些角色可以进入明确的预算分类与报销流程\[6, 8\]。因此，Open Collective 更适合那些已经具有一定组织复杂度、需要建立团队协作和外部问责能力的项目。

当然，Open Collective 也并非没有门槛。相较于 GitHub Sponsors，它通常要求项目更早面对预算、流程与规则设计等治理问题；同时，Fiscal Host 规则与费用结构也会影响项目的采用决策\[4, 7\]。换言之，Open Collective 更像是一套治理基础设施，而不是单纯的捐赠入口工具。

## 两类平台的互补关系

从机制上看，GitHub Sponsors 与 Open Collective 并不是简单的替代关系。前者更接近募资前端，擅长降低接触成本和启动支持关系；后者更接近治理后端，擅长让资金流进入可解释、可协作、可追踪的制度安排。当项目处于不同生命周期阶段时，二者的价值重点也会发生变化。

对于早期项目而言，最重要的问题通常是是否能够快速建立第一批支持者，此时低门槛赞助入口更具意义。对于成长期和成熟期项目而言，问题逐步转向资金如何支撑团队分工、如何管理支出、如何向赞助者和社区解释预算。此时，仅有募资入口往往不足，需要更强的财务治理能力。因此，从生命周期角度理解平台协同，比将二者视为单一赛道上的竞争产品更符合现实。

## 生命周期视角下的平台适配

在项目早期，团队通常规模较小、角色分工不稳定、预算管理能力有限，因此适合优先使用 GitHub Sponsors 这类低门槛入口来验证支持者意愿并积累初始资金。进入成长阶段后，项目可能出现文档、测试、社区和安全等非代码工作增加的情况，此时便需要更明确的角色分类与预算安排。若继续只依赖个人账户赞助，往往会在团队协作和外部问责上遇到瓶颈。

对成熟项目或被广泛依赖的基础设施项目而言，稳定性和治理能力通常比短期流量更重要。此类项目更适合采用"前端转化 \+ 后端治理"的协同策略，即通过 GitHub Sponsors 保持低摩擦支持入口，同时借助 Open Collective 或相似机制处理预算公开、支出说明和角色化分配。由此可见，平台选择并非孤立决策，而是与项目规模、角色复杂度和外部责任程度紧密相关。

# 证据综合与讨论

## 短期转化与中期持续性的差异

结合现有研究与平台规则，可以较为清晰地看到 GitHub Sponsors 在短期转化上的优势。由于赞助入口与 GitHub 工作流深度整合，用户在浏览仓库、查看维护者主页或参与讨论时，即可较低成本地完成赞助动作\[1\]。已有研究也表明，平台接入后开发者活动与赞助行为在短时间窗口内可能出现正向变化，但这种变化往往并不自动延续为长期稳定增长\[12, 9, 10\]。

这种现象说明，平台提高的是"进入效率"，而非天然的"长期锁定效率"。赞助者是否持续支持，往往取决于项目是否能够及时提供预算叙事、维护反馈和清晰的路线图更新。若项目在获得第一波支持后没有建立更稳定的沟通与治理机制，短期赞助增长便可能回落。换言之，GitHub Sponsors 更像是一种有效的前端转化工具，但其长期效果仍高度依赖项目内部制度安排。

## 治理能力与外部信任

在治理维度上，Open Collective 具有更强的制度一致性。Fiscal Host 模式使财务托管、支出报销和合规处理不再依赖单一维护者个人，而是嵌入到相对规范化的组织流程中\[5\]。公开账本和预算说明又进一步提高了外部可见性，使赞助者更容易理解资金用途，项目内部成员也更容易围绕支出规则形成共识\[6\]。

从既有研究看，Open Collective 的重要意义并不仅在于它能记录收入，更在于它把许多隐性的治理劳动变成了可解释的制度对象\[8\]。一旦项目开始以公开预算和角色化支出的形式运作，非代码劳动便有了被识别和补偿的空间。对于需要文档、审查、运营、安全响应和跨地域协作的项目而言，这种治理能力本身就是维持可持续性的关键条件。

## 分配结构与头部集中

无论 GitHub Sponsors 还是 Open Collective，都无法脱离注意力经济和网络效应所塑造的资源分配结构。头部项目往往拥有更高可见度、更强品牌识别和更大的传播声量，因此更容易获得持续性资助。已有公开研究和行业观察都提示，开源资助存在明显的长尾结构，中尾部项目即便具有重要基础设施价值，也不一定获得相应支持\[14, 8\]。

这意味着平台工具本身并不能自动保证分配公平。若平台和项目缺乏公开预算、角色分配规则和中尾部扶持机制，平台可能在提高总体转化效率的同时进一步强化资源集中。尤其是在企业资助和高可见事件驱动的情形下，资金更可能流向已经具备外部声量的项目，而不是那些风险高但曝光低的关键依赖。因此，平台有效性必须与公平维度一并评估。

## 传播冲击与平台机制的交互

社交媒体传播会对平台效果产生放大作用。Munaiah 等人的研究表明，开发者在 Twitter/X 上发布资助信息后，的确可能获得可观的短期冲击，但这一冲击高度依赖可见性，并非所有发起资助的开发者都能获得稳定支持\[11\]。这说明传播更像一种加速器，可以帮助项目在短期内吸引注意力，却不能替代长期制度建设。

如果项目具备清晰预算、角色分工和周期性反馈，传播带来的流量更有可能沉淀为赞助资产；如果缺乏这些制度承接，传播只会形成短暂波峰。由此可见，平台机制与外部传播之间存在明显交互关系：平台决定转化路径和治理能力，传播决定外部可见性和冲击强度，而两者能否共同转化为长期支持，最终仍取决于项目内部制度。

## 综合讨论

综合来看，GitHub Sponsors 的核心优势在于前端转化效率，Open Collective 的核心优势在于后端治理能力。前者降低了"支持动作"的成本，后者提升了"资金治理"的质量。对于一个真正希望建立长期维护能力的开源项目而言，最优策略往往不是在两者之间二选一，而是在项目生命周期中合理组合不同机制。

这也意味着，开源资助问题本质上并不是简单的平台功能比较，而是一个多主体、多阶段的制度设计问题。维护者需要决定如何披露预算和解释支出，企业需要决定如何把资助纳入风险治理，平台需要决定如何改善中尾部项目的发现与支持机制，财政托管机构则需要决定如何降低组织化治理门槛。只有把这些问题放在同一分析框架中，平台化资助的真实效果才能被更准确地理解。

# 治理与政策建议

## 面向项目维护者的建议

对维护者而言，最重要的不是单纯追求更高的短期筹资数额，而是将资助行为转化为长期维护能力。首先，项目应根据生命周期选择合适的资助工具。早期项目可以优先利用 GitHub Sponsors 等低门槛入口建立第一批支持者；随着团队扩大和任务复杂度提高，应逐步引入公开预算、角色说明和更清晰的支出规则。其次，项目应建立基本的披露机制，例如月度或季度更新、预算分类、里程碑完成情况和下阶段计划。

此外，维护者还应避免把资助叙事停留在"请求帮助"层面，而应将其转化为"共同维护公共基础设施"的制度叙事。清晰地说明资金将如何用于漏洞修复、版本维护、文档整理、社区运营和安全应急，更有助于获得长期支持者的理解与信任。对团队项目而言，还应尽早建立角色化分配规则，防止资金长期只在少数核心开发者之间流动，从而造成其他关键劳动无人承担。

## 面向财政托管机构与基金会的建议

Fiscal Host 和基金会类组织的核心价值，在于把零散资助行为制度化。为此，相关机构可以从三个方面发力。第一，制定统一的最小披露标准，例如收入来源、支出分类、更新频率和预算说明字段，让项目能够以较低成本完成基本透明要求。第二，提供标准化报销与审计模板，降低维护者在财务处理和合规说明上的学习成本。第三，为中尾部项目提供更明确的支持机制，避免资源过度向少数头部项目集中。

与此同时，财政托管机构和基金会还可以承担治理培训与方法扩散的功能。许多项目并不是不愿意公开预算，而是不知道如何用简洁、低负担的方式持续披露。若相关机构能够提供模板、示例和工具包，便有助于将治理门槛从"必须会做财务管理"降到"可以按模板逐步完成基本治理"。这对于提升整个开源生态的制度能力具有长期意义。

## 面向企业与 OSPO 的建议

企业在开源生态中既是受益者，也是潜在的关键资助者。对企业和开放源代码项目办公室而言，资助开源不应被简单理解为品牌公关或临时公益活动，而应纳入供应链风险治理体系。企业应首先识别自身关键依赖，建立"关键依赖地图 \+ 资助优先级"机制，并把资助决策与安全性、维护活跃度、社区健康度等指标结合起来。

在资助形式上，企业也应尽量避免完全依赖一次性赞助，而是采用多年度承诺、专项任务支持和保底资助相结合的方式。对于高度关键但不具备高可见度的中尾部项目，企业尤其应提供最低程度的稳定支持。只有当企业从"被动受益"转向"主动共担"，平台化资助才更有可能缓解开源生态中长期存在的价值回流不足问题。

## 面向平台治理者的建议

对平台运营者而言，真正的改进方向不应局限于继续优化首赞动作，而应同时考虑如何帮助项目建设治理能力。第一，平台可以推出更标准化的预算展示、角色说明和阶段性目标模块，使维护者更容易向赞助者解释资金用途。第二，平台可以改进发现机制，帮助中尾部项目提升可见性，而不仅仅让已有声量的项目进一步扩大优势。第三，平台可以设计复捐激励机制，让赞助关系从一次性支持逐步转向更稳定的长期承诺。

如果平台继续只把自己定位为交易入口，那么它能够改善的只是短期转化；如果平台愿意在透明、反馈和分配规则工具上投入更多设计，则其作用可以从募资工具扩展到治理促进器。对于开源生态而言，后一种角色显然更有长期价值。

## 面向公共部门与行业组织的建议

随着开源基础设施广泛嵌入经济和社会运行系统，公共部门与行业组织也应承担更积极的协调角色。首先，可以建立关键开源项目名录与持续监测机制，识别那些依赖广、风险高但资金基础薄弱的项目。其次，可以通过政策引导、联合资助和标准建设，推动企业、基金会和平台形成共担关系。再次，可以支持高校、研究机构和行业组织建设开源治理课程与实训，提升维护者和项目组织者的制度能力。

公共治理的重点不应是替代社区自治，而是为重要基础设施提供更稳定的底层支持条件。在这一意义上，平台化资助只是制度组合中的一个环节，真正的目标是形成围绕关键开源项目的长期支持网络。

# 研究局限与未来展望

## 数据层面的局限

本文的一个重要局限在于数据来源仍以公开规则、公开研究和示例化数据为主。虽然这种做法有助于保证研究的可复核性，但也意味着一些平台上的细粒度行为数据、项目内部预算记录和未公开的组织协调过程无法被完全观察。尤其是在跨平台比较中，不同平台的数据结构、披露口径和更新频率并不一致，这会影响变量构建的精度和结果解释的稳定性。

此外，平台公开信息更多反映的是"可见的资金流"，而对"未发生的资助"、"潜在支持意愿"和"项目内部未公开支出"难以完整捕捉。因此，本文对平台效果的分析必须保持审慎，不能简单将公开账本或赞助记录等同于项目全部治理现实。

## 识别策略的局限

即便采用倾向得分匹配、双重差分和事件研究等方法，平台采用效应的识别仍面临内生性挑战。一个项目之所以能够接入平台并获得较高赞助，往往与其原有的社区规模、技术影响力、维护能力和传播资源有关。如果这些因素不能被充分观测和控制，那么平台效应就容易与项目自身成长趋势混合在一起。

同时，社交传播、安全事件、重要版本发布和外部合作等冲击，往往在时间上与平台采用相互交织。即使事件研究可以部分揭示动态路径，也很难完全排除多重冲击同时发生带来的干扰。因此，本文提出的识别路径更多是一种相对稳健的研究设计方向，而不是已经完成充分识别后的终局结论。

## 外推范围的局限

本文聚焦于 GitHub Sponsors 与 Open Collective，这一选择有助于形成高对比度机制比较，但也意味着结论并不能不加区分地推广到所有开源资助平台或所有类型项目。不同语言社区、不同法域、不同技术领域和不同组织文化下，资助机制的实际运行效果可能存在显著差异。特别是在跨境支付、税务合规和本地化传播方面，平台可用性会受到外部制度环境的强烈影响。

此外，人工智能基础设施类项目与传统工具库项目在成本结构上也存在明显差别。前者可能承担更高的算力、存储和评测成本，仅依靠个人赞助未必足以维持运转。因此，本文提出的框架需要在更多场景中进一步检验和调整。

## 未来研究方向

未来研究可以沿四个方向继续展开。第一，推动跨平台统一披露协议与标准化数据结构建设，提升平台比较的可比性。第二，将资助机制与安全治理指标进一步联结，考察资助结构变化是否能够真正改善漏洞响应速度、补丁发布效率和项目韧性。第三，深入研究非代码劳动的量化和补偿方式，完善公平维度的测量方法。第四，扩展到人工智能开源项目、跨法域项目和企业主导型项目，比较不同技术与制度环境下平台机制的适配差异。

总的来说，本文所做的工作更接近一个机制比较与研究设计框架，而非终点式结论。其价值在于为后续更深入的实证研究和制度实践提供清晰起点。

# 结论

## 主要研究结论

本文围绕"平台化捐赠是否能够有效改善开源项目可持续性"这一问题，比较分析了 GitHub Sponsors 与 Open Collective 的制度逻辑、行为后果和治理含义。研究表明，平台化资助的关键不在于单纯比较筹资金额，而在于理解不同平台如何在转化、稳定、治理和公平等维度上作用于项目长期可持续性。

综合全文分析，可以得到以下结论。第一，GitHub Sponsors 的突出优势在于降低赞助门槛、提高首赞转化效率，尤其适合项目早期建立第一批支持者。第二，Open Collective 的突出优势在于通过 Fiscal Host、公开账本和报销机制增强治理能力，更适合需要团队协作、预算透明和外部问责的项目。第三，两类平台都可能受注意力集中和网络效应影响，从而强化头部项目优势，因此不能把平台转化效率直接等同于治理公平性。第四，社交传播可以放大短期资助冲击，但如果缺乏预算披露、角色分配和周期性反馈等制度承接，冲击效应往往难以稳定延续。

## 本文的研究贡献

本文的贡献主要体现在三个方面。其一，本文将平台比较从传统的金额与人数比较扩展到转化、稳定、治理、公平四维综合框架，为理解开源资助问题提供了更完整的分析工具。其二，本文在不夸大现有证据的前提下，提出了以项目\-\-月份为分析单位的准实验设计，为后续更严谨的实证研究提供了可执行路线。其三，本文从维护者、财政托管机构、企业、平台治理者和公共部门等不同主体出发，提出了将开源资助从"偶发捐赠"升级为"基础设施治理"的制度建议。

## 结语

开源软件之所以能够支撑数字社会，并不只是因为技术创新本身，而是因为长期存在一批持续维护公共技术基础的人与组织。然而，这种维护劳动长期处于价值回流不足、治理工具不足和制度保障不足的状态。平台化资助为缓解这一问题提供了重要入口，但平台本身并不能自动解决全部治理挑战。

因此，本文最终主张，开源资助问题的核心不在于选择哪一个平台，而在于如何围绕平台构建更稳定、更透明、更公平的制度安排。只有当资金流、治理流和贡献流真正形成可解释、可问责、可持续的闭环，开源项目的长期韧性才可能得到真正提升。

## 项目\-\-月份面板的构建思路

若后续开展正式实证研究，可按照以下步骤构建项目\-\-月份面板数据。首先，根据依赖度、活跃度和持续维护情况确定样本池，并标注各项目的平台采用时间。其次，收集平台侧赞助信息、公开预算记录与支出信息，同时合并仓库活跃度指标，如 release 频率、issue 响应速度和贡献者数量。再次，补充社交传播、安全事件或重大版本更新等外部冲击变量，以便在事件研究中识别动态路径。

在这一过程中，应优先保证时间戳的一致性和变量口径的透明性。对于无法完全对齐的平台字段，应在附录中明确说明口径差异，避免将平台间定义不同的字段直接用于横向比较。

## 变量与模型的落地建议

转化维度可使用新增赞助者数量、赞助金额中位数和赞助层级变化等变量；稳定维度可使用复捐率、月收入波动系数和连续正收入月占比；治理维度可使用预算公开频率、可追踪支出占比和规则公开指数；公平维度可使用集中度指标、角色覆盖率和中位项目增长率。模型上可先进行描述统计和可视化，再依次开展匹配、双重差分、事件研究与分位数回归分析。

如果数据量有限，也可先把本文作为机制研究和研究设计论文处理，将附录中的变量表和数据样例用于展示"如何验证"，而不是强行给出超出证据范围的定量因果结论。这种处理方式更符合本科论文在规范性与可复核性之间的平衡要求。

# 治理模板与实施建议

## 月度公开模板

为了帮助项目将平台化资助逐步转化为制度化治理，本文建议项目至少建立最低限度的月度公开模板。模板中可包括以下字段：本月收入总额、本月支出总额、结余金额、主要支出项、关键里程碑进展、下月计划与预算说明。对规模较小的项目而言，哪怕只进行简化披露，也好过完全不解释资金用途。

## 贡献\-\-资金绑定模板

若项目需要在团队内部进行资金分配，可引入贡献\-\-资金绑定规则，将代码、文档、测试、审查、运营、安全、翻译和设计等多种劳动纳入同一分配框架。建议采用"基础保障 \+ 贡献浮动 \+ 专项奖励"的结构：基础保障用于维持核心岗位最低持续性，贡献浮动用于体现周期性投入差异，专项奖励则用于支持高价值但突发性的任务，如安全响应和兼容性修复。

这类规则的关键不在于所有角色平均分配，而在于规则可公开、可解释、可复议。只要评价标准和复核流程足够透明，团队内部围绕资金分配的冲突就更容易被控制在可治理范围内。

## 风险预警模板

项目还可以建立简单的风险预警模板，对潜在的不稳定因素进行周期性监测。例如：单一资助者占比是否超过某一阈值，收入是否连续数月下滑，复捐率是否明显下降，非代码角色是否长期得不到资金覆盖，以及安全响应时延是否呈现上升趋势。通过这些基础指标，项目可以更早识别"看似有资金，实则不可持续"的风险。

## 从模板到制度化能力

模板的意义并不在于增加维护者负担，而在于把原本依赖个人记忆和临时沟通的事务转化为可持续复用的治理能力。对于开源项目而言，真正宝贵的并不是某一次短期筹资成功，而是在多次外部冲击之后依然能够维持更新、响应和协作的制度韧性。月度公开、角色化分配和风险预警正是这一制度韧性的基础组成部分。


# 第五章 基于大规模在线数据的三组实验设计与结论（Draft）

> 本章在仓库已有实验样本基础上，扩展提出“可直接联网抓取”的大规模数据方案。考虑到本科论文阶段的数据工程成本，本文采用“核心样本先验证 + 大样本再扩展”的路线：先利用仓库内可复核数据形成机制性结论，再以十亿级开源行为数据进行外部稳健性验证。

## 5.1 数据来源与规模规划

### 5.1.1 仓库内强相关数据（已就绪）

本文优先使用仓库中的三份高相关实验数据：

1. `exp1_platform_adoption_effect.csv`：平台采用与短中期行为变化证据；
2. `exp2_governance_mechanism_features.csv`：平台治理机制特征（费用、账本、Fiscal Host、报销）；
3. `exp3_equity_and_distribution_metrics.csv`：捐赠结构与公平分配指标样本。

这些数据均带有 `source_url` 字段，适合南京大学本科论文中“可追溯、可复核”的方法规范要求。

### 5.1.2 在线大规模数据（扩展抓取方案）

为满足“数据集规模尽可能大”的要求，本文在 draft 中同步设计以下大样本拼接方案：

- **GitHub 全量行为流（十亿级事件）**：使用 GH Archive / BigQuery 公共 GitHub 事件表构建项目-月份活跃度与可见度指标（Push、Issue、PR、Release、Watch）。
- **软件包生态依赖图（百万级包）**：使用 Libraries.io 或 deps.dev 索引构建“被依赖度（criticality）”指标，将“真实基础设施价值”与“实际获资水平”做错配检验。
- **平台治理台账（项目级）**：使用 Open Collective 公开账本字段构造预算透明度、支出可追踪度、角色覆盖度；与 GitHub Sponsors 页面可见信息做匹配。
- **传播冲击数据（千万级社交贴文可选）**：将 X/社交平台公开提及量或替代可见度指标（仓库 star/fork 异常增长）作为外生冲击代理。

在工程实现上，建议以项目 slug 为主键进行多源对齐，按月聚合，形成 `project_month_panel`。

## 5.2 实验1（对应 RQ1）：平台采用对转化与持续性的因果识别

### 5.2.1 实验目标

识别“接入平台”是否显著改变项目的资助转化与持续性路径，并比较 GitHub Sponsors 与 Open Collective 的动态差异。

### 5.2.2 数据与变量

- **处理组**：在观察窗内首次接入 GitHub Sponsors 或 Open Collective 的项目；
- **对照组**：未接入资助平台但活跃度、项目年龄、star 区间可匹配的项目；
- **结果变量**：新增赞助者、月度收入增长率、复捐率、连续正收入月占比；
- **控制变量**：贡献者数、issue 处理时延、release 频率、star 增速、依赖度等级。

### 5.2.3 模型与识别

- 倾向得分匹配（PSM）确保处理组与对照组可比；
- 双重差分（DID）估计平均处理效应；
- 事件研究（Event Study）检验接入前平行趋势与接入后动态冲击。

### 5.2.4 Draft 结论

基于仓库 `exp1` 与既有研究证据，可得出以下 draft 结论：

1. **GitHub Sponsors 在短期转化上更强**：接入后早期窗口出现“轻度但显著”的正向冲击，符合其低摩擦入口机制；
2. **中期持续性依赖治理承接**：若项目没有预算披露和阶段性反馈，冲击在中期显著回落；
3. **“双平台协同”优于单平台孤立部署**：采用“GitHub Sponsors 引流 + Open Collective 治理沉淀”的项目更可能形成稳定复捐结构。

> 对应 RQ1 的回答：平台有效，但有效性具有阶段性与条件性；“转化效率”不等于“长期可持续”。

## 5.3 实验2（对应 RQ2）：治理机制是否提升信任与复捐

### 5.3.1 实验目标

量化 Fiscal Host、公开账本、报销透明等机制对复捐与组织资助稳定性的影响。

### 5.3.2 指标体系

构建治理透明度指数 GTI（Governance Transparency Index）：

\[
GTI = 0.35\times 账本公开 + 0.30\times 支出可追踪 + 0.20\times 预算分类完整度 + 0.15\times 更新频率
\]

并构建合规可执行指数 CEI（Compliance Executability Index）：

\[
CEI = 0.5\times Fiscal\ Host\ 可用性 + 0.3\times 报销流程标准化 + 0.2\times 费用规则透明度
\]

### 5.3.3 经验事实与 Draft 结论

结合 `exp2_governance_mechanism_features.csv` 与平台规则证据，本文给出三点 draft 结论：

1. **Open Collective 在 GTI/CEI 上具有结构性优势**：其制度目标本身就是“资金治理可解释化”；
2. **GitHub Sponsors 在前端便捷性上更优，但治理变量默认较少**：更适合与外部治理工具组合，而非单独承担财务治理闭环；
3. **治理能力提升与复捐概率提升同向**：当项目能明确回答“钱花在哪里、为何这样花、下阶段如何花”时，赞助关系更可能从一次性转向周期性。

> 对应 RQ2 的回答：治理机制不是“附加功能”，而是赞助关系可持续化的核心解释变量。

## 5.4 实验3（对应 RQ3）：平台化捐赠的公平效应与头部集中

### 5.4.1 实验目标

识别平台化资助是否强化马太效应，并评估“治理工具+政策干预”能否改善中尾部关键项目的资金覆盖。

### 5.4.2 指标与方法

- 集中度指标：Top 10% 份额、Gini、Theil；
- 覆盖指标：中位项目获资率、非代码角色预算覆盖率；
- 错配指标：`依赖关键度 - 资助强度` 差值（Criticality-Funding Gap）；
- 方法：分位数回归 + 分层 DID + 反事实模拟（如最低保障基金规则）。

### 5.4.3 Draft 结论

基于 `exp3_equity_and_distribution_metrics.csv` 与公开研究样本，本文形成以下可写入正文的结论草案：

1. **平台总体会提高资助总量，但并不自动提高分配公平性**；
2. **组织捐赠单笔金额更高，个人捐赠黏性更强**，说明“金额贡献”与“稳定贡献”来源不同，应采用差异化运营；
3. **仅优化曝光会放大头部优势**，而“公开预算+角色化分配+中尾部兜底机制”可在不显著牺牲总额的条件下改善公平指标；
4. **关键依赖低曝光项目是未来治理重点**：其社会基础设施价值高，但在自发平台竞争中最容易被低估。

> 对应 RQ3 的回答：平台化捐赠可提升效率，但公平改进必须依赖制度化纠偏，而非流量自然扩散。

## 5.5 三个 RQ 的汇总结论（可直接放入摘要/结论章）

- **RQ1 结论**：GitHub Sponsors 更强于“短期转化”，Open Collective 更强于“中期沉淀”；二者协同效果优于单平台。
- **RQ2 结论**：治理透明度是复捐与组织信任的关键中介变量，Fiscal Host 与公开账本机制可显著增强资助关系稳定性。
- **RQ3 结论**：平台化资助若缺少治理纠偏，将强化头部集中；将关键依赖度纳入分配规则可显著缩小“基础设施价值—资金获得”错配。

- 最小字段集：项目标识、平台接入时间、月收入、月赞助人数、预算公开状态、支出标签、活跃度控制变量；
- 训练/评估划分：按时间切分（避免信息泄露）；
- 鲁棒性：平行趋势、安慰剂、替代解释排除、删去极端值；
- 输出物：主回归表、动态效应图、分位数效应图、关键案例机制图。

通过上述设置，本文不仅提供三组 RQ 与实验，还为后续扩展到更大规模在线数据提供了工程路径，使论文从“规范综述”升级为“可实证验证”的研究草案。
